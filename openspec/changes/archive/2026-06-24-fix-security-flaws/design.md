## Context

During a security review, two major flaws were discovered:
1. **Voucher Validation Flaw**: The `vouchers.ts` route has a logic block that grants immediate enrollment to any user who submits a code starting with `VTC-`, completely bypassing database validation.
2. **Missing Admin Authorization**: The `admin.ts` routes use a global `authenticate` middleware, which only validates that the user possesses a valid JWT token. It fails to check if the user actually has the `admin` role, leaving critical admin-only endpoints (e.g., editing plans, altering system settings like PagSeguro API tokens) accessible to any logged-in user, such as regular students.

## Goals / Non-Goals

**Goals:**
- Eliminate the unauthorized bypass logic in voucher redemption.
- Ensure that every endpoint in `server/src/routes/admin.ts` enforcing admin privileges properly checks for the `admin` role.
- Maintain existing functionality for legitimate users and administrators.

**Non-Goals:**
- Rewrite the entire authentication or authorization system.
- Implement granular permission checks for routes that only require basic `admin` access (we will stick to the existing `requirePermission` or role-based check for the scope of this fix).

## Decisions

### 1. Removing Voucher Bypass
**Decision**: Delete the block of code in `server/src/routes/vouchers.ts` that automatically approves any code starting with `VTC-`.
**Rationale**: Real B2B vouchers generated by the platform do start with `VTC-`, but they are always stored in the database. Removing the bypass forces the code to fall through to the legitimate database validation logic that checks if the `voucher` exists and has an `active` status.

### 2. Enforcing Admin Role on All Admin Routes
**Decision**: Create a global middleware check in `server/src/routes/admin.ts` (e.g., `requireAdmin` or using `requirePermission`) to enforce that the `jwtUser.role` is `admin`.
**Rationale**: Instead of manually adding `requirePermission` to every single route definition in `admin.ts` (which is error-prone and caused the current bug), we will apply an admin-role-check middleware globally alongside `authenticate` to the `/admin/*` path. 

*Alternative Considered*: Add `requirePermission(...)` to each individual route. This is rejected because it risks future developers forgetting to add the middleware to new admin routes. A blanket policy on `/admin/*` is safer.

## Risks / Trade-offs

- **Risk**: Applying a global admin check on `/admin/*` might break routes that are currently under `/admin/` but are legitimately accessed by non-admins (e.g., a file upload proxy for students, if any).
  - **Mitigation**: Review all routes defined in `admin.ts` to ensure none are intended for student use. Based on current inspection, all routes under `admin.ts` are strictly for platform administrators. The `/uploads/:filename` route does not use the `/admin/` prefix, so it won't be affected.
